# AGENTS.md

이 파일은 Wiki의 백과사전이 아니라 **짧은 지도와 작성 계약**이다. 자세한 지식은 `docs/`에 두고, 여기에는 모든 문서에 공통으로 적용할 원칙만 남긴다.

기준 철학: [OpenAI, Harness Engineering](https://openai.com/ko-KR/index/harness-engineering/)

## 1. Wiki의 목적

이 Wiki는 다음 목적을 동시에 가진 공개 기록 시스템이다.

- 개인 Second Brain과 장기 학습 기록
- 공식 자료를 읽고 검증한 기술 리뷰
- 질문과 시행착오를 설계 판단으로 바꾸는 과정 기록
- 실제 프로젝트와 연결되는 설계 문서
- 면접관이 조사력, 판단력, 학습 과정, 실무 연결 능력을 확인할 수 있는 포트폴리오

완성된 결론만 남기지 않는다. 처음의 질문, 잘못된 가설, 정정 근거, 관점 변화와 최종 판단을 함께 보존한다.

## 2. 하니스 엔지니어링 원칙

- 사람은 목표와 판단 기준을 정하고, 에이전트는 조사·작성·검증을 수행한다.
- 좋은 결과는 모델 이름만이 아니라 컨텍스트, 도구, 제약, 관측 가능성, 피드백 루프에서 나온다.
- 컨텍스트는 희소 자원이다. `AGENTS.md`를 거대한 설명서로 만들지 않는다.
- 저장소를 기록 시스템으로 사용한다. 채팅에만 있는 중요한 판단은 문서로 옮긴다.
- 처음에는 짧은 지도를 제공하고 필요한 문서로 점진적으로 안내한다.
- 문서만으로 유지하기 어려운 규칙은 링크 검사, 형식 검사 등 기계적 검증으로 옮긴다.
- 오래된 주장과 중복 문서는 주기적으로 정리한다. 문서 드리프트를 방치하지 않는다.
- 모든 것을 세세히 지시하지 않는다. 사실성, 출처, 구조 같은 불변 조건을 정하고 표현 방식에는 자율성을 둔다.

## 3. 저장소 지도

- `README.md`: 독자를 위한 짧은 소개와 문서 탐색
- `TODO.md`: 세부 구현 추적이 아닌 간단한 학습·프로젝트 목표
- `docs/`: 장문 기술 문서와 설계 기록
- `docs/ai-agent-systems/`: AI 에이전트, 하니스, 오케스트레이션 관련 글
- `docs/ai-foundations/`: 임베딩, 벡터 검색 등 독립적으로 읽는 AI·ML 기초 개념 글
- `docs/learning-roadmaps/`: 장기 학습 목표와 단계별 실행 계획
- `docs/learning-evidence/`: 학습자의 원문 답변, 정정 과정, 놓친 개념을 보존하는 학습 증거와 작성 규칙

새 디렉터리를 만들면 이 지도에 필요한 진입점만 추가한다.

학습 증거를 작성하거나 수정할 때는 반드시 `docs/learning-evidence/README.md`를 먼저 읽고 따른다. 학습자의 답변은 요약하거나 교정하지 않고 원문 그대로 보존하며, AI의 피드백은 구조화해 놓친 부분과 기억해야 할 판단 기준을 중심으로 기록한다.

## 4. 문서 작성 계약

장문 문서는 가능한 한 다음 흐름을 따른다.

1. 독자가 이해할 질문과 문제 상황
2. 용어의 정의와 등장 배경
3. 초기 가설 또는 기존 접근
4. 공식 자료와 실제 경험에 근거한 원인 분석
5. 관점이 바뀐 과정과 설계 선택
6. 구체적인 예시, 다이어그램 또는 비교표
7. 결과, 트레이드오프, 한계
8. 현재 시점의 결론과 배운 점

초심자가 읽을 수 있도록 새 용어는 **정의 → 배경 → 기존 방식과 차이 → 예시** 순서로 설명한다.

장문 아티클은 제목 아래에 한 줄짜리 메타데이터 표를 두고, 본문이 시작되기 전에 주요 `##` 절로 이동하는 목차를 둔다. 메타데이터 표와 목차 사이에는 글의 핵심을 보여 주는 한 줄 리드만 둘 수 있다. 목차에는 본문 탐색에 필요한 상위 절만 넣고 세부 소제목까지 나열하지 않는다.

## 5. 근거와 주장

- 공식 문서와 1차 자료를 우선한다.
- 제품 기능은 작성 시점의 공식 문서에서 다시 확인한다.
- 출처가 이동했으면 현재 공식 URL과 확인 날짜를 기록한다.
- 사실, 설계 해석, 추천, 미검증 가설을 같은 수준의 주장처럼 섞지 않는다.
- 수치와 계산은 출처 또는 재현 가능한 계산 방법을 함께 남긴다.
- 확인하지 못한 기능, API, 파일 경로, 구현 상태를 사실처럼 쓰지 않는다.
- 대화 중 판단이 정정됐다면 가장 최신 결론을 따르고 정정 이유를 보존한다.

권장 표기:

- **공식 기능**: 1차 출처가 직접 확인하는 내용
- **설계 해석**: 여러 자료를 종합한 작성자의 판단
- **추천**: 특정 상황에 대한 선택
- **미검증 가설**: 실험이나 실제 적용이 더 필요한 주장

## 6. 예시와 다이어그램

- 실행 가능한 예시는 실제로 검증한 문법만 사용한다.
- 검증하지 않은 구조는 `text` 코드 블록과 함께 의사코드라고 명시한다.
- 공급자 문서에 없는 YAML 필드, API 이름, 출력 형태를 추측해 만들지 않는다.
- Mermaid는 장식이 아니라 상태, 경계, 소유권 또는 흐름을 설명할 때만 사용한다.
- 보안과 권한을 다루는 예시는 비신뢰 입력, 최소 권한, 검증 경계와 실패 경로를 함께 설명한다.

## 7. 문체

- 한국어로 명확하고 직접적으로 쓴다.
- 과장된 중요성, 근거 없는 전망, 기계적인 전환 문구를 피한다.
- 같은 결론을 여러 표현으로 반복하지 않는다.
- 긴 서론보다 구체적인 질문과 사례로 시작한다.
- 표와 목록은 비교나 탐색을 쉽게 만들 때만 사용한다.
- 개인적인 학습 과정은 남기되 감상보다 근거와 판단 변화를 중심에 둔다.
- 각 절은 결론이나 핵심 정의를 먼저 제시하는 두괄식으로 쓴다.
- 증상, 구성 요소, 판단 기준, 선택지는 `- **항목:** 설명` 형태로 계층화한다.
- 줄글은 생각의 전환, 원인과 결과의 연결, 자세한 근거를 설명할 때 사용한다.
- 모든 문장을 목록으로 바꾸지 않는다. 목록 사이에 짧은 줄글을 두어 탐구 과정과 판단 변화가 이어지게 한다.

## 8. 메타데이터

모든 아티클에 메타데이터를 넣되 독자의 읽기를 방해하지 않도록 작게 유지한다.

front matter의 기본 필드는 다음 일곱 개다.

```yaml
---
title: "문서 제목"
description: "한 문장 설명"
author: yjs000
published: YYYY-MM-DD
updated: YYYY-MM-DD
reading_time: 약 N분
tags: [tag-one, tag-two]
---
```

본문 제목 아래에는 다음처럼 한 줄 표로 보여 준다.

```markdown
| 작성자 | 게시·수정일 | 읽는 시간 | 태그 |
|---|---|---|---|
| yjs000 | YYYY-MM-DD | 약 N분 | Tag One · Tag Two |
```

- 읽는 시간은 front matter와 코드 블록을 제외한 본문 단어 수를 분당 약 200단어로 계산해 반올림한다.
- 게시일과 수정일이 같으면 날짜 하나만 표시한다. 다르면 `게시 YYYY-MM-DD · 수정 YYYY-MM-DD`로 구분한다.
- 상세 출처는 본문의 `참고 자료`, 판단 변화는 본문 서사에 둔다.
- review 상태, 긴 대상 독자 목록, scope 목록, changelog, decision log를 front matter에 쌓지 않는다. Git 이력과 본문이 그 역할을 맡는다.
- 특별한 검색 요구가 있을 때만 필드를 추가하고, 기본값으로 되돌릴 수 있으면 추가하지 않는다.

Wiki는 구현 상태 저장소가 아니다. 문서 자체의 상태와 실제 프로젝트 존재 여부만 기록하고, 세부 구현 진행률·작업 ID·증거 파일을 추적하지 않는다.

기술을 설명하기 위한 schema, 상태 모델, 의사코드, 검증·복구 예시는 장문 문서에 둘 수 있다. 이는 학습과 설계 분석을 위한 예시이며, 실제 프로젝트의 진행 상태나 구현 증거로 취급하지 않는다.

## 9. Related Projects와 TODO

- `Related Projects`에는 실제로 존재하며 현재 문서와 직접 관계있는 프로젝트만 적는다.
- 프로젝트마다 어떤 관계인지와 어떤 도구를 활용했는지 짧게 설명한다.
- 가상의 저장소, `TBD` 프로젝트, 미래 구현 상태를 Related Projects에 넣지 않는다.
- 현재 관련 프로젝트는 `yjs000/law-rag`이며 프로젝트 진행에 Codex를 활용했다.
- `TODO.md`에는 큰 학습 목표만 둔다.
- 목표와 연결되는 실제 프로젝트가 생겼을 때만 체크하고 프로젝트 링크를 남긴다.
- TODO에 세부 작업, 의존성 그래프, 완료 증거 규약, 구현 진행률을 만들지 않는다.

## 10. README 원칙

README는 소개와 탐색에 집중한다.

- Wiki 설명
- 문서 링크
- 학습 목표 링크
- 현재 Related Projects

문서 작성 원칙은 README에 중복하지 않고 이 파일에서만 관리한다.

## 11. 변경 절차

1. `AGENTS.md`, `README.md`, 관련 문서를 먼저 읽는다.
2. 직접 제공된 원문과 공식 출처를 우선 확인한다.
3. 기존 문서의 주장과 최신 자료가 충돌하는지 찾는다.
4. 문서를 수정하면서 중복 링크와 오래된 상태 표현을 함께 정리한다.
5. front matter, 내부 링크, 외부 링크, 코드 펜스와 Mermaid 구조를 검사한다.
6. README, TODO, Related Projects가 서로 모순되지 않는지 확인한다.
7. Git diff를 검토하고 의미가 분명한 커밋을 만든다.
8. 원격에 반영했다면 로컬과 원격 SHA, 주요 파일을 다시 확인한다.

## 12. 완료 기준

- 독자가 문제와 결론뿐 아니라 판단이 바뀐 이유를 이해할 수 있다.
- 핵심 주장은 공식 출처 또는 명시된 경험에 연결된다.
- 초심자도 용어와 비교 기준을 따라갈 수 있다.
- 실행 예시는 검증됐거나 의사코드로 분명히 표시됐다.
- README에는 원칙이 중복되지 않는다.
- Related Projects에는 실제 프로젝트만 존재한다.
- TODO는 간단한 학습 목표로 유지된다.
- 깨진 내부 링크와 명백한 문서 드리프트가 없다.
