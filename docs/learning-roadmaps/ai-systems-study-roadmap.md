## 목표와 현재 조건

### 최종 목표

목표는 특정 프레임워크를 능숙하게 쓰는 사람이 되는 것이 아니다. 새로운 AI 제품이나 오픈소스를 접했을 때 다음 과정을 독립적으로 수행할 수 있는 AI 시스템 설계자가 되는 것이다.

1. 입력이 어디로 들어오는지 찾는다.
2. 요청이 어떤 상태를 거치는지 추적한다.
3. 어떤 모델과 도구가 어느 조건에서 호출되는지 확인한다.
4. 실패 지점, 재시도, 중단 후 복구 방식을 찾는다.
5. 해당 구조를 선택한 이유와 비용을 설명한다.
6. 요구사항에 필요한 부분과 불필요한 추상화를 구분한다.
7. 작은 비교 실험을 만들고 채택 여부를 근거로 결정한다.
8. 검색, 모델, 런타임, 하네스, 평가, 배포를 하나의 시스템으로 설계한다.

최종적으로는 다음과 같은 판단을 코드와 실험을 근거로 말할 수 있어야 한다.

- 이 시스템의 병목은 모델보다 상태 관리에 있다.
- 이 루프는 상태를 메모리에만 두기 때문에 재시작에 취약하다.
- 이 멀티에이전트 구조는 실제 병렬 실행보다 프롬프트 역할 분담에 가깝다.
- 이 기능은 프레임워크 전체를 도입하지 않고 큐와 상태 머신으로 구현할 수 있다.
- 이 모델은 품질이 가장 높은 모델이 아니라 현재 평가셋의 비용·지연·근거 충실도 조건을 가장 잘 만족한다.

### 다섯 가지 학습 목표

1. 일반 검색과 RAG 검색의 차이를 알고 요구사항에 맞는 검색 시스템을 설계한다.
2. 작업별 평가셋을 만들고 적절한 모델을 선택한다.
3. 오픈소스의 구현 로직과 알고리즘을 읽고 비교한다.
4. AI가 안정적으로 작업하도록 컨텍스트, 도구, 검증, 피드백을 설계하는 하네스 엔지니어링을 한다.
5. 툴을 연결해 중단, 재시도, 승인, 검증, 재개가 가능한 지속 실행 환경을 설계한다.

### 현재 위치

현재는 AI 오픈소스 생태계에 대한 관심과 문제의식은 높지만, 구현을 코드 기준으로 이해하는 능력은 시작 단계다.

- **강점:** 새로운 개념의 큰 구조를 빠르게 파악한다.
- **강점:** 기능 목록보다 도구의 역할, 중복 계층, 오버헤드, 실패 지점을 질문한다.
- **강점:** API, DB, 인증, Docker, 배포, 로그 등 기존 웹 개발 경험을 AI 시스템에 연결할 수 있다.
- **약점:** 프로젝트의 진입점과 한 요청의 실행 흐름을 직접 추적한 경험이 적다.
- **약점:** 검색 알고리즘, 모델 특성, Python AI 생태계, 상태 머신과 작업 큐는 새로 쌓아야 한다.
- **병목:** 하나를 완성하기 전에 관심사가 다음 도구와 더 큰 구조로 확장된다.
- **병목:** 학습 대상의 오류보다 Next.js, Supabase, 환경 변수, 배포, 타입 오류를 해결하는 데 시간이 더 많이 든다.
- **병목:** 검증 결과를 읽고 직접 원인을 가설화하기보다 다음 수정 프롬프트를 작성하는 흐름이 반복된다.

현재 자기평가인 오픈소스 구현 이해도 1~5%를 출발점으로 삼는다. 도구 이름과 역할을 아는 것과 소스코드의 실행 흐름 및 설계 선택을 평가하는 것은 별개의 능력이다.

### 시간 조건

- **2026년 7월 22일~8월 18일:** 주당 20시간, 총 약 80시간
- **2026년 8월 19일 이후:** 주당 4시간, 월 약 16~18시간
- **첫 3개월 총량:** 약 117시간
- **첫 달 이후 주간 기본 배분:** 구현·실험 2시간, 코드 읽기 1시간, 공식 자료 확인 30분, 회고·기록 30분

시간이 줄어든 뒤에는 여러 프로젝트를 병렬로 진행하지 않는다. 에너지 법령 RAG를 공통 실험장으로 유지하고 검색, 모델 평가, 하네스를 차례로 덧붙인다.

### 목표 기간

현재 시간 조건을 유지할 때의 기준이다. 시간만 채우는 것이 아니라 각 단계의 달성 조건을 통과해야 다음 단계로 이동한다.

- **1년:** 작은 AI 시스템의 주요 실행 흐름을 읽고 수정한다.
- **2년:** 주요 검색·에이전트 구현 패턴을 코드와 실험으로 비교한다.
- **3년:** 요구사항에 맞춰 검색, 모델, 하네스, 런타임을 상당 부분 독립적으로 설계한다.
- **최신 기술 자립:** 약 2년이면 새 기술을 기존 구조와 비교할 수 있고, 약 3년이면 제품명이 바뀌어도 스스로 학습 경로를 만들 수 있다.

모든 모델과 프레임워크 소식을 따라가는 것은 목표가 아니다. 검색과 컨텍스트 구성, 모델 호출, 툴 실행, 상태 저장, 작업 분해, 검증, 재시도, 관찰 가능성, 권한, 비용 제한처럼 오래 남는 구성요소를 먼저 익힌다.

## 로드맵의 순서

학습 순서는 다음과 같이 고정한다.

```text
RAG 데이터 흐름
→ 검색 알고리즘
→ 모델 선택과 평가
→ 오픈소스 읽기와 축소 재구현
→ 하네스 엔지니어링
→ 단일 에이전트 런타임
→ 지속 실행 시스템
→ 멀티에이전트와 독립 설계
```

### 에이전트 구현을 뒤로 미루는 이유

에이전트 런타임을 먼저 만들면 모델 호출과 툴 호출을 배우기 전에 환경 설정, 비동기 처리, 상태 저장, 재시도, UI, 배포 오류가 한꺼번에 들어온다. 지금의 학습 패턴에서는 이 복잡성이 프롬프트 작성과 무지성 오류 수정 시간을 다시 늘릴 가능성이 높다.

하네스를 먼저 공부한다는 것은 에이전트 런타임을 모른 채 추상적인 이론만 읽는다는 뜻이 아니다. 이미 사용 중인 Codex와 저장소를 대상으로 다음을 설계한다.

- 에이전트에게 보여 줄 파일과 규칙
- 작업 크기와 완료 조건
- 실행 가능한 검증 명령
- 실패 로그와 다음 시도에 넘길 정보
- 사람이 승인해야 하는 경계
- 테스트, 린트, 타입 검사 피드백

이 경험을 쌓은 다음 최소 에이전트 루프를 직접 구현하면, 각 기능이 왜 필요한지 이미 알고 시작할 수 있다.

### 단계 통과 원칙

각 단계는 다음 네 가지 증거가 모두 있어야 끝난다.

1. **설명:** 그림을 보지 않고 입력부터 출력까지 자신의 말로 설명한다.
2. **실험:** 100줄 안팎의 작은 코드로 핵심 동작 하나를 재현한다.
3. **비교:** 최소 두 가지 방법을 같은 입력과 지표로 비교한다.
4. **적용:** 통과한 방법만 에너지 법령 RAG 또는 다음 학습 저장소에 반영한다.

프로젝트가 실행된다는 사실만으로 통과하지 않는다. Codex가 작성한 설명을 읽고 이해한 느낌이 드는 것도 통과가 아니다.

## 첫 3개월 상세 계획

첫 3개월은 3주 단위 네 블록과 마지막 8일의 전환 구간으로 나눈다. 첫 블록은 에너지 법령 RAG MVP를 끝내는 집중 구간이고, 이후에는 검색, 모델 평가, 오픈소스 읽기의 순서를 지킨다. 이 기간에는 에이전트 런타임과 멀티에이전트를 구현하지 않는다.

### 1블록: 2026년 7월 22일~8월 11일 — RAG 전체 경로를 작게 완성

**사용 가능 시간:** 약 60시간

**핵심 목표:** 에너지 법령 RAG MVP를 배포하고, 법령 원문이 답변의 근거가 되기까지의 전체 경로를 코드와 DB를 보며 설명한다.

#### 7월 22일: 구현 전 개념 게이트 — 2시간

1. AI에게 “20분 안에 이해할 만큼만” RAG의 전체 구조를 묻는다.
2. 답변을 닫고 종이나 빈 문서에 다음 흐름을 직접 그린다.

```text
법령 원문
→ 파싱
→ 조문 단위 청킹
→ 임베딩
→ 벡터 저장소
→ 질문 임베딩
→ 유사도 검색
→ 검색 문맥 구성
→ 모델 답변
→ 근거 조문 표시
```

3. 각 화살표에 “입력”, “출력”, “저장 위치”, “실패하면 보이는 증상”을 한 줄씩 쓴다.
4. AI에게 자신의 설명을 검증시킨다. 틀린 부분만 고친다.
5. 다음 질문에 답하지 못하면 프로젝트 생성으로 넘어가지 않는다.
   - 문서를 왜 청킹하는가?
   - 임베딩은 무엇을 비교 가능하게 만드는가?
   - 검색 결과와 모델 답변은 왜 따로 평가해야 하는가?
   - 법령에서 조문 번호와 제목을 메타데이터로 남겨야 하는 이유는 무엇인가?

#### 7월 23~24일: 세 개의 독립 실험 — 4시간

프로젝트를 만들지 말고 각각 독립 실행되는 작은 실험으로 확인한다.

- **실험 A — 청킹:** 텍스트 파일 하나를 조문 또는 제목+본문 단위로 나누고 출력한다.
- **실험 B — 임베딩:** 두 문장의 임베딩과 유사도 값을 출력한다.
- **실험 C — 검색:** 문서 조각 10개를 저장한 뒤 질문 하나의 상위 3개 결과와 점수를 출력한다.

각 실험 전에는 예상 결과를 먼저 적는다. 실행 후 예상과 다른 이유를 한 문장으로 기록한다. 환경 오류가 20분을 넘으면 전체 프로젝트를 고치지 말고 실험 환경만 새로 만든다.

#### 7월 25일~8월 2일: 에너지 법령 RAG MVP — 29시간

MVP는 다음 범위로 고정한다.

**반드시 포함할 기능**

- 법령 문서 3~10개 수집 및 원문 저장
- 조문 또는 제목+본문 단위 청킹
- 조문 번호, 법령명, 제목을 포함한 메타데이터
- 임베딩 생성과 pgvector 저장
- 질문 입력과 유사 문서 검색
- 검색 문맥을 이용한 답변 생성
- 답변 아래 근거 조문 표시
- 원문 확인 경로
- 대표 질문 20개
- 배포와 README의 데이터 흐름 그림

**제외할 기능**

- 법령 전체 자동 동기화와 개정 감지
- 리랭커와 복잡한 하이브리드 검색
- 인증과 관리자 화면
- 법령 diff와 체크리스트 자동 생성
- 다수 모델 자동 라우팅
- 에이전트 및 멀티에이전트
- 완전 자동 평가 시스템

**시간 상한**

| 작업 | 상한 |
|---|---:|
| 기본 구조와 DB 연결 | 3시간 |
| 데이터 확보·정리 | 5시간 |
| 조문 청킹 | 3시간 |
| 임베딩·벡터 저장 | 4시간 |
| 검색 API | 3시간 |
| 답변·출처 연결 | 4시간 |
| 최소 UI | 2시간 |
| 배포·환경 변수 | 2시간 |
| 대표 질문 검증·수정 | 3시간 |
| README와 구조 그림 | 2시간 |

시간 상한을 넘기면 기능을 추가하지 않고 원인을 `학습 대상`, `프로젝트 환경`, `도구 사용` 중 하나로 분류한다. 학습 대상이 아닌 환경 오류는 해결 과정을 길게 공부하지 않고 재현 조건과 해결 결과만 남긴다.

#### 8월 3~6일: 실행 경로 역추적 — 12시간

완성된 프로젝트를 새로 고치기보다 자신이 구현한 시스템을 읽는다.

1. 사용자 질문이 들어오는 함수부터 시작한다.
2. 호출되는 함수와 변하는 데이터를 한 단계씩 기록한다.
3. 다음 경로를 실제 파일명과 함수명으로 다시 그린다.
4. 각 단계에서 입력과 출력의 예시 한 개를 붙인다.
5. Codex가 생성한 코드를 줄 단위로 모두 읽으려 하지 말고, 한 요청의 경로만 끝까지 추적한다.

달성 조건은 다음 질문에 저장소를 보며 답하는 것이다.

- 법령은 정확히 어떤 규칙으로 잘리는가?

- 유사도 검색은 애플리케이션과 DB 중 어디에서 수행되는가?
- 검색된 조문은 어떤 형식으로 프롬프트에 들어가는가?
- 답변의 출처 링크는 어떤 메타데이터에서 만들어지는가?
- 검색 실패와 생성 실패를 로그에서 어떻게 구분하는가?

블록 종료 시에는 기능을 더 넣지 않고 다음 세 문서를 남긴다.

#### 4개월차

- Python 패키지 구조, 의존성, 설정 로딩 방식 읽기
- CLI 또는 HTTP 진입점 찾기
- 디버거와 로그로 한 요청 추적
- 모듈 지도를 1페이지로 제한해 작성

#### 5개월차

- 검색 또는 모델 어댑터 하나 분석
- 상태 객체가 생성되고 변하는 지점 기록
- 오류가 상위 계층으로 전달되는 방식 확인
- 핵심 경로를 작은 코드로 재구현

#### 6개월차

- 두 구현을 동일 과제로 비교
- 추상화 비용, 교체 가능성, 오류 처리, 테스트 용이성 비교
- 현재 에너지 법령 RAG에 한 가지 설계만 시험 적용
- 적용 전후 결과와 복잡도 기록

**달성 조건**

- README 없이도 진입점과 주요 실행 경로를 찾을 수 있다.
- 코드 전체를 읽지 않고 현재 질문에 필요한 범위를 정할 수 있다.
- 원본 구현과 축소 구현의 차이를 설명한다.
- 기능표가 아니라 파일, 함수, 상태 변화, 실행 결과를 근거로 비교한다.

### 7~9개월차: 저장소 하네스 엔지니어링

**목표:** Codex가 에너지 법령 RAG 저장소에서 범위를 지키며 변경, 검증, 수정, 완료 보고를 반복하게 한다.

#### 7개월차 — 컨텍스트와 작업 계약

- 저장소의 짧은 지도와 핵심 규칙
- 작업 단위의 크기 제한
- 수정 가능한 경로와 금지 경로
- 완료 조건과 검증 명령
- 필요한 문서만 점진적으로 찾는 구조

#### 8개월차 — 피드백 루프

- 테스트, 린트, 타입 검사, 빌드 명령 연결
- 실패 로그를 다음 시도에 전달
- 검색 평가셋 회귀 검사
- 모델 응답의 구조화 출력 검사
- 부분 성공을 완료로 처리하지 않는 규칙

#### 9개월차 — 관찰 가능성과 승인

- 작업 시작, 변경, 검증, 실패, 재시도 기록
- 비용과 최대 시도 횟수 제한
- 외부 배포와 데이터 변경 전 사람 승인
- 완료 근거와 남은 부채를 분리한 보고
- 중단 후 다음 세션이 이어받을 수 있는 짧은 상태 기록

**달성 조건**

Codex가 하나의 제한된 작업을 받아 다음 순서를 지키며 수행한다.

```text
작업 읽기
→ 관련 컨텍스트 찾기
→ 변경
→ 테스트
→ 실패 원인 분류
→ 수정
→ 전체 검증
→ 완료 근거와 남은 부채 기록
```

### 10~12개월차: 하네스 반복 개선과 두 번째 오픈소스 분석

**목표:** 하네스가 실제 실패를 줄였는지 측정하고, 다른 구현의 검증·컨텍스트 관리 방식과 비교한다.

- 하네스 적용 전후의 재시도 횟수와 실패 유형 비교
- 잘못된 파일 수정, 테스트 누락, 범위 확장 사례 기록
- 규칙을 문서로 둘지 기계 검사로 옮길지 결정
- 오래된 규칙과 중복 컨텍스트 정리
- 두 번째 오픈소스에서 컨텍스트, 툴, 상태, 검증 경로 분석
- 내 하네스에 필요한 한 가지 개선만 반영

**달성 조건**

- 규칙 파일이 길어진 것이 아니라 검증 성공률이나 오류 발견 시점이 개선됐다.
- 실패 사례를 근거로 하네스 변경 이유를 설명한다.
- 모델 성능 문제와 하네스 문제를 구분한다.
- 새 도구를 추가하지 않고도 기존 피드백 루프를 개선한 사례가 있다.

### 13~18개월차: 최소 단일 에이전트 런타임

**목표:** 프레임워크 없이 최소 에이전트 루프를 구현하고 신뢰성 기능을 한 개씩 추가한다.

#### 13~15개월차 — 최소 루프

- 메시지와 툴 정의 전달
- 모델의 툴 호출 파싱
- 허용된 툴만 실행
- 툴 결과를 메시지에 추가
- 모델 재호출
- 종료 조건과 최대 스텝
- 구조화 출력 검증

#### 16~18개월차 — 신뢰성

- 타임아웃과 제한된 재시도
- 실행 로그와 trace ID
- 체크포인트와 세션 저장
- 중단 후 재개
- 비용과 토큰 상한
- 사람 승인 단계
- 동일 작업 중복 실행 방지

**달성 조건**

- 프로세스를 의도적으로 중단한 뒤 저장된 체크포인트에서 재개한다.
- 허용되지 않은 툴 호출을 차단한다.
- 최대 스텝과 비용 제한이 실제로 동작한다.
- 실패 이유를 로그만으로 찾을 수 있다.
- LangGraph, OpenAI Agents SDK 등 하나의 원본 구현에서 같은 기능이 어디에 있는지 찾는다.

### 19~24개월차: 지속 실행 시스템

**목표:** Discord 또는 GitHub에서 받은 작업이 DB에 저장되고, 워커 중단 후에도 재개되며, 검증을 통과한 작업만 완료되게 한다.

- 작업 큐와 워커
- 명시적 상태 머신
- 체크포인트와 이벤트 로그
- 락과 동시성 제어
- 멱등성 키
- 재시도 가능한 오류와 불가능한 오류 구분
- 실패 작업 격리와 수동 재처리
- 승인 흐름
- 비용·시간·스텝 제한
- 모니터링과 알림

**달성 조건**

- 같은 이벤트가 두 번 들어와도 결과가 중복 생성되지 않는다.
- 워커가 중단돼도 작업 상태가 사라지지 않는다.
- 재시도 횟수를 넘긴 작업이 무한 루프에 빠지지 않는다.
- 사람이 승인해야 할 변경은 승인 전 외부에 반영되지 않는다.
- 테스트와 평가를 통과한 작업만 완료 상태가 된다.

### 25~30개월차: 오픈소스 비교와 채택 판단

**목표:** 에이전트 프레임워크 3개와 RAG 구현 2개를 동일 과제로 비교한다.

비교 항목은 다음과 같다.

- 상태 저장 위치
- 실행 흐름의 통제 주체
- 체크포인트와 재개 단위
- 툴 등록과 권한 경계
- 병렬 처리 방식
- 구조화 출력과 검증
- 실패 전파와 재시도
- 관찰 가능성
- 도입 시 추가되는 운영 복잡도
- 프레임워크를 사용하지 않을 때 직접 구현할 범위

**달성 조건**

- 동일 입력, 동일 평가 기준, 동일 운영 조건으로 비교한다.
- 데모 성공과 장기 운영 가능성을 구분한다.
- GitHub 스타와 마케팅 문구를 채택 근거로 쓰지 않는다.
- 특정 요구사항에서는 가장 유명한 프레임워크를 채택하지 않는 결론도 설명할 수 있다.

### 31~36개월차: 독립적인 AI 시스템 설계

**목표:** 새로운 요구사항을 받고 검색, 모델, 하네스, 런타임, 평가, 배포를 하나의 설계로 만든다.

- 요구사항과 실패 비용 분석
- RAG가 필요한 문제와 필요하지 않은 문제 구분
- 검색 전략과 데이터 갱신 설계
- 모델 평가와 라우팅 기준
- 에이전트가 필요한 작업과 일반 워크플로로 충분한 작업 구분
- 상태, 큐, 체크포인트, 승인 경계
- 하네스와 자동 검증
- 로그, 추적, 비용 통제
- 권한과 샌드박스
- 장애 복구와 운영 절차

**최종 달성 조건**

새 AI 제품이나 오픈소스를 접한 뒤 1~2주 안에 다음을 수행한다.

1. 구조와 주요 실행 경로를 파악한다.
2. 핵심 구현과 상태 경계를 찾는다.
3. 작은 재현 또는 비교 실험을 만든다.
4. 품질, 비용, 실패, 운영 복잡도를 평가한다.
5. 자신의 시스템에 채택할지 결정한다.
6. 선택하지 않은 대안이 더 적합한 조건도 설명한다.

## 공부 방식

### 기존 방식에서 바꿀 점

기존 흐름은 다음과 같았다.

```text
새 개념 발견
→ 큰 구조 질문
→ 바로 프로젝트 설계
→ Codex 구현
→ 실행 실패
→ 수정 프롬프트
→ 다시 실패
→ 다시 수정
→ 뒤늦게 개념 이해
```

이 방식은 실행력은 높지만, 학습 대상과 무관한 오류가 전체 시간을 차지한다. 프롬프트와 오류 수정에 60~70%, 개념 이해에 30~40%가 쓰이는 구조다.

앞으로는 다음 흐름으로 고정한다.

```text
새 개념 발견
→ 20분 개요
→ AI를 닫고 직접 개념도 작성
→ 자신의 말로 설명
→ 설명 검증
→ 질문형 노트
→ 100줄 이하의 작은 실험
→ 예상과 실제 비교
→ 프로젝트에 한 조각만 반영
→ 같은 평가셋으로 검증
→ 회고와 다음 가설
```

핵심 원칙은 다음 한 문장이다.

> 새로운 기술을 발견하면 바로 프로젝트를 만들지 않는다. 먼저 30분 안에 설명할 수 있을 정도까지 이해하고, 100줄 이하의 작은 실험으로 검증한 뒤 프로젝트에 넣는다.

### 한 개념을 공부하는 10단계 사이클

#### 1. 질문을 한 문장으로 제한한다

“RAG를 공부한다”가 아니라 “법령 검색에서 조문 단위 청킹이 문단 고정 길이 청킹보다 근거 회수에 유리한가?”처럼 쓴다. 한 세션에서 질문은 하나만 다룬다.

#### 2. AI에게 20분 분량의 개요만 요청한다

용어 목록을 늘리지 않는다. 다음 네 항목만 받는다.

- 정의
- 등장한 문제
- 입력과 출력
- 없을 때 생기는 실패

#### 3. AI를 보지 않고 개념도를 그린다

화살표마다 데이터 형태와 담당 구성요소를 쓴다. 그림이 그려지지 않으면 아직 구현하지 않는다.

#### 4. 3분 설명을 녹음하거나 글로 쓴다

다음 문장으로 시작한다.

- 이 기능이 해결하는 문제는 …
- 입력은 …이고 출력은 …이다.
- 이 기능이 없으면 …가 실패한다.
- 비슷한 방법과 다른 점은 …이다.

#### 5. AI에게 설명의 오류만 검증시킨다

처음부터 새 설명을 생성하게 하지 않는다. 자신의 설명을 입력하고 사실 오류, 빠진 전제, 혼동한 용어만 지적하게 한다.

#### 6. 질문형 노트를 만든다

기술명만 제목으로 쓰지 않는다.

- 왜 문서를 청킹해야 하는가?
- 왜 조문 번호를 메타데이터로 저장하는가?
- 언제 벡터 검색보다 키워드 검색이 유리한가?
- 검색은 맞았는데 답변이 틀릴 수 있는 이유는 무엇인가?

답은 먼저 기억으로 쓰고, 그다음 자료를 확인해 수정한다.

#### 7. 100줄 이하의 독립 실험을 설계한다

프로젝트 DB, UI, 인증, 배포를 붙이지 않는다. 입력과 출력이 터미널에서 바로 보이게 만든다. 한 실험은 한 가설만 검증한다.

#### 8. 실행 전에 예상 결과를 적는다

예상 순위, 실패 조건, 출력 형태를 먼저 적는다. 실행 후 결과가 다르면 코드를 바로 고치기 전에 가설이 틀렸는지, 구현이 틀렸는지, 입력 데이터가 부적절한지 분류한다.

#### 9. 프로젝트에는 통과한 한 조각만 반영한다

작은 실험이 통과해도 프로젝트에 넣는 순간 통합 오류가 생길 수 있다. 한 번에 하나만 반영하고 기존 평가셋을 다시 실행한다.

#### 10. 회고에서 다음 질문 하나만 정한다

새 도구 목록을 만들지 않는다. 이번 실험에서 가장 큰 불확실성 하나를 다음 세션의 질문으로 넘긴다.

### AI와 Codex를 쓰는 규칙

- **설명 전에 답 생성 금지:** 먼저 자신의 설명이나 예상 결과를 쓴다.
- **수정 전에 진단:** 오류 메시지, 재현 단계, 예상 동작, 실제 동작, 첫 가설을 함께 제공한다.
- **한 프롬프트 한 변경:** 여러 오류와 리팩터링을 한 번에 요청하지 않는다.
- **검증 명령 요구:** 수정 결과와 함께 실행한 테스트, 실제 출력, 남은 실패를 받는다.
- **코드 경로 직접 확인:** 학습문서만 읽지 않고 실제 파일과 함수를 연다.
- **부분 성공 금지:** 실행됐다는 말보다 평가셋과 완료 조건을 확인한다.
- **새 도구 즉시 도입 금지:** 현재 단계의 가설을 더 싸게 검증할 수 있을 때만 도입한다.

### 디버깅이 학습을 삼키지 않게 하는 규칙

오류가 발생하면 먼저 분류한다.

| 분류 | 예시 | 처리 방식 |
|---|---|---|
| 학습 대상 오류 | 청킹 경계가 조문을 깨뜨림, 검색 순위가 예상과 다름 | 원인 가설을 세우고 충분히 분석한다. |
| 통합 오류 | 데이터 형식 불일치, API 계약 오류 | 최소 재현으로 축소하고 한 경계만 고친다. |
| 환경 오류 | 환경 변수, 배포, 패키지 버전 | 20분 상한을 두고 해결 결과만 기록한다. |
| 도구 사용 오류 | 잘못된 프롬프트, 너무 큰 작업 요청 | 작업을 줄이고 완료 조건을 다시 쓴다. |

20분 동안 증거가 늘지 않으면 같은 수정 프롬프트를 반복하지 않는다. 다음 순서를 따른다.

1. 오류를 재현하는 최소 명령을 만든다.
2. 마지막으로 성공한 단계와 처음 실패한 단계를 나눈다.
3. 입력, 출력, 로그를 저장한다.
4. 가설을 하나만 선택한다.
5. 그 가설을 반증할 가장 작은 실험을 실행한다.
6. 학습 대상과 무관한 환경 문제라면 우회가 아니라 범위 축소 또는 세션 종료를 선택한다.

### 주당 4시간 운영 방식

#### 첫 2시간 — 구현 또는 실험

- 현재 블록의 질문 하나
- 예상 결과 기록
- 작은 실험 실행
- 프로젝트 반영은 최대 한 변경

#### 다음 1시간 — 코드 읽기

- 실제 파일과 함수 기록
- 이해하지 못한 주변 코드는 보류

#### 다음 30분 — 공식 자료와 동향

- 현재 블록과 직접 관련된 공식 문서 또는 저장소 릴리스만 확인
- 최신 제품 뉴스 수집 금지
- 새 도구는 아래 다섯 질문에만 답하고 보류 여부 결정

1. 어느 계층을 담당하는가?
2. 기존에 직접 구현해야 했던 무엇을 대신하는가?
3. 상태는 어디에 저장되는가?
4. 실패하면 어떻게 복구하는가?
5. 도입하면 어떤 복잡성이 추가되는가?

#### 마지막 30분 — 회고와 기억 점검

- 자료를 닫고 배운 내용을 5분간 작성
- 질문형 노트 1~3개 갱신
- 시간 배분 기록
- 다음 주 질문 하나 결정

시간 비율은 다음 네 분류로만 기록한다.

- A: 개념 설명과 기억 점검
- B: 작은 실험과 직접 코드 읽기
- C: 프로젝트 통합과 평가
- D: 프롬프트 작성, 대기, 환경 오류, 반복 수정

학습 방식이 바뀌었다고 판단하는 기준은 A+B+C가 늘고 D가 줄었다는 느낌이 아니다. 4주 연속으로 다음 조건을 만족해야 한다.

- D가 전체 시간의 35% 이하다.
- 모든 프로젝트 변경 전에 작은 실험 또는 기존 평가 증거가 있다.
- 오류 수정 요청의 80% 이상에 자신의 가설이 포함된다.

- 자료 없이 설명하는 기억 점검을 매주 수행한다.

## 범위가 다시 커지는 것을 막는 규칙

### 기능 추가 규칙

새 기능은 다음 질문에 모두 `예`일 때만 현재 블록에 넣는다.

1. 현재 블록의 달성 조건에 직접 필요한가?
2. 100줄 안팎의 독립 실험으로 먼저 확인했는가?
3. 기존 평가셋으로 좋아졌는지 판정할 수 있는가?
4. 추가 환경과 도구가 한 개 이하인가?
5. 이번 주의 다른 목표를 제거해도 우선할 가치가 있는가?

하나라도 `아니오`면 보류 목록에 둔다.

### 새 도구 채택 규칙

- 현재 구성요소 중 무엇을 대체하는지 말할 수 있어야 한다.
- 상태와 실패 복구 위치를 코드 또는 공식 문서에서 확인해야 한다.
- 기존 방식과 동일 과제로 비교해야 한다.
- 도입으로 늘어나는 의존성, 배포, 관찰 비용을 적어야 한다.
- 데모가 아니라 현재 평가셋의 결과가 개선돼야 한다.

### 학습 완료로 착각하지 않는 규칙

다음은 완료 증거가 아니다.

- AI가 좋은 학습문서를 만들어 줬다.
- 튜토리얼을 끝까지 따라 했다.
- 프로젝트가 한 번 실행됐다.
- 유명 프레임워크를 연결했다.
- 자동 judge가 높은 점수를 줬다.
- 오픈소스의 README와 아키텍처 그림을 읽었다.

완료 증거는 자신의 설명, 작은 재현, 같은 조건의 비교, 실제 코드 경로, 실패 사례, 프로젝트 검증 결과다.
