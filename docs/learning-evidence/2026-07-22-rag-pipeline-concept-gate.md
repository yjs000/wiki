## 학습 맥락

이 기록은 [AI 시스템 학습 로드맵의 1블록 구현 전 개념 게이트](../learning-roadmaps/ai-systems-study-roadmap.md#1블록-2026년-7월-22일8월-11일--rag-전체-경로를-작게-완성)에 대한 첫 개념 검증 기록이다. 법령 원문이 답변의 근거가 되기까지 각 단계의 입력, 출력, 저장 위치, 실패 증상을 설명하고 네 개의 필수 질문에 답했다.

- **현재 증거 수준:** 개념 설명
- **아직 없는 증거:** 독립 실행 청킹·임베딩·검색 실험, 프로젝트 코드·테스트를 통한 적용 확인
- **법령 예시의 성격:** 학습자가 제공한 전기사업법 제7조 예시를 사용했으며, 이 문서에서 최신 법령 원문 여부는 별도로 검증하지 않았다.

## 학습자 원문

### RAG 단계별 첫 설명

아래 내용은 학습자가 처음 작성한 표현을 수정하거나 요약하지 않고 그대로 옮겼다.

```text
법령 원문

→ 파싱
“입력” : 원문데이터
“출력” : 원문의 조각. (정규화된 원문)
“저장 위치” : DB의 원문 테이블. (파싱된 원문을 DB에 저장해도 되는건가)
“실패하면 보이는 증상” : 저장되지 않음. 로그가 남음.


→ 청킹

“입력” : 파싱된 원문 데이터
“출력” : 원문의 조각.(원문의 조각이 정확히 뭐지..) 형태소들. 조문, 시점, 제목, 키워드, 등의 메타데이터.
“저장 위치” : DB. 청킹 테이블.
“실패하면 보이는 증상” : 저장된 원문으로 파싱을 다시함.


→ 임베딩

“입력” : 청킹
“출력” : 벡터
“저장 위치” : db 벡터 테이블.
“실패하면 보이는 증상” : 저장된 청크로 다시 임베딩.
여기서 AI가 왜 사용되는지 모르겠음.

→ 벡터 저장소

“입력” : 임베딩된 벡터데이터
“출력” : X
“저장 위치” : db 벡터 테이블.
“실패하면 보이는 증상” : 임베딩부터 저장을 다시함.

→ 질문 임베딩

“입력” : 사용자 질문
“출력” : 질문을 임베딩하여 벡터로 출력
“저장 위치” : 질문 이력의 벡터정보 테이블.
“실패하면 보이는 증상” : 질문 이력만 쌓이고 시스템 로그로 실패 정보가 남음.


→ 유사도 검색

“입력” : 임베딩된 사용자 질문과 저장된 벡터데이터들
“출력” : 질문과 저장된 데이터의 유사도를 검색하여 topk를 출력.
“저장 위치” : 질문이력의 유사도 topk테이블
“실패하면 보이는 증상” : 문맥에 맞는 결과값이 나오지 않음.


→ 검색 문맥 구성

“입력” : 유사도 topk결과와 임베딩된 질문
“출력” : 유사도 topk결과, 임베딩한 질문, 관련 원문 index위치
“저장 위치” : 저장하지 않고 모델에게 반환.
“실패하면 보이는 증상” : 문맥에 맞는 결과값이 나오지 않음.

→ 모델 답변

“입력” : 검색문맥
“출력” : 사용자 친화적인 답변과 근거데이터
“저장 위치” : 질문이력테이블에 답변으로 저장.
“실패하면 보이는 증상” : 문맥에 맞는 결과값이 나오지 않음.

→ 근거 조문 표시

“입력” : 모델답변과 근거 위치데이터
“출력” : 근거 원문
“저장 위치” : 저장하지 않음.
“실패하면 보이는 증상” : 근거 원문을 볼 수 없음
```

### 개념 게이트 네 질문에 대한 답변

아래 답변도 학습자가 작성한 원문 그대로다.

```text
문서를 왜 청킹하는가? : 임베딩의 효율을 위해?
임베딩은 무엇을 비교 가능하게 만드는가? : 질문과 유사한 근거데이터를 비교할 수 있게 한다.
검색 결과와 모델 답변은 왜 따로 평가해야 하는가? : 검색결과로 유사한 topk가 나왔다고 하더라도, 모델이 답변을 잘못 추론할 수 있다.
법령에서 조문 번호와 제목을 메타데이터로 남겨야 하는 이유는 무엇인가? : 조금 더 정확한 근거의 계층화를 위해서
```


## 현재 판정

네 질문 가운데 임베딩의 비교 대상과 검색·답변 분리 평가는 핵심 방향을 이해했다. 청킹의 목적과 법령 메타데이터의 역할은 일부만 설명되어 보완이 필요했다.

| 질문 | 판정 | 이해한 부분 | 보완할 핵심 |
|---|---|---|---|
| 문서를 왜 청킹하는가 | 부분 이해 | 임베딩 처리와 관련 있다고 보았다. | 핵심 목적은 효율 자체보다 검색·인용 단위를 만들고 제한된 모델 문맥에 필요한 근거만 넣는 것이다. |
| 임베딩은 무엇을 비교 가능하게 만드는가 | 이해 | 질문과 근거 데이터의 유사성을 비교한다고 설명했다. | 텍스트의 의미를 같은 차원의 숫자 벡터로 바꾼다는 중간 표현을 기억한다. |
| 검색과 답변을 왜 따로 평가하는가 | 이해 | 검색이 맞아도 모델 답변이 틀릴 수 있음을 구분했다. | 검색 실패인데 답변만 그럴듯한 반대 사례도 함께 구분한다. |
| 조문 번호와 제목을 왜 남기는가 | 부분 이해 | 근거의 계층을 보존하려 했다. | 주된 목적은 안정적인 식별, 정확한 인용, 원문 이동과 사용자 검증이다. |

## 놓친 부분과 정정

### 파싱 결과와 청크는 서로 다른 데이터다

초기 설명에서는 파싱과 청킹의 출력이 모두 `원문의 조각`으로 표현됐다. 두 단계는 분리해야 한다.

- **파싱 출력:** 법령의 조·항·호 구조, 제목, 본문을 기계가 다룰 수 있는 형태로 정규화한 문서다. 아직 검색 단위를 결정한 것은 아니다.
- **청킹 출력:** 검색 결과로 반환하고 모델에 근거로 전달할 단위다. 법령에서는 짧은 조, 항, 또는 상위 문맥이 붙은 호가 될 수 있다.
- **형태소와의 차이:** 형태소는 언어 분석을 위한 최소 의미 단위다. 청크는 사람이 읽고 출처로 확인할 수 있는 검색·근거 단위이므로 형태소처럼 잘게 자르지 않는다.

이 구분이 없으면 파싱 실패와 청킹 규칙 실패를 같은 문제로 다루게 된다. 예를 들어 `제5항`이 누락된 것은 파싱 문제지만, `제5항 제1호`가 상위 허가기준 문장 없이 저장된 것은 청킹 문제다.

### 청킹의 중심 목적은 임베딩 비용 절감이 아니다

임베딩 모델의 입력 길이와 처리 비용도 청킹에 영향을 준다. 그러나 법령 RAG에서 먼저 기억할 목적은 검색 정확도와 근거 보존이다.

문서 전체를 하나의 벡터로 만들면 허가 요건, 변경 절차, 예외처럼 서로 다른 의미가 하나의 좌표에 섞인다. 반대로 구절 단위로 지나치게 자르면 주체·의무·조건·예외의 관계가 끊어진다. 좋은 청크는 다음 조건을 동시에 만족해야 한다.

- 질문 하나와 비교할 수 있을 만큼 주제가 좁다.
- 단독으로 읽었을 때 주체, 행위, 조건의 의미가 남는다.
- 답변의 근거로 표시했을 때 사용자가 이해할 수 있다.
- 법령명·조문 번호·제목을 통해 원문으로 돌아갈 수 있다.

따라서 청킹은 `작게 자를수록 좋다`가 아니라 `검색할 수 있을 만큼 작고, 의미를 잃지 않을 만큼 크게 유지한다`는 경계 선택이다.

### 임베딩은 의미 비교를 가능하게 하지만 정답을 판정하지 않는다

임베딩 모델은 질문과 청크의 텍스트를 같은 차원의 숫자 벡터로 바꾼다. 의미가 비슷하다고 학습된 텍스트가 벡터 공간에서 가까워지므로 코사인 유사도나 거리로 순위를 계산할 수 있다.

벡터가 가깝다는 사실은 `관련될 가능성이 높다`는 뜻이지, 해당 조문이 법적 정답이라는 뜻은 아니다. 개정 시점이 다르거나, 비슷한 용어를 쓰는 다른 법령이거나, 예외 조항이 빠졌을 수 있다. 임베딩은 후보 검색 수단이고 정답 검증기는 아니다.

### 원래 질문과 질문 벡터의 사용처가 다르다

초기 검색 문맥 설명에는 `임베딩된 질문`을 모델에 반환한다고 적었다. 질문 벡터는 유사도 계산에만 사용한다. 생성 모델이 답변할 때 받는 것은 숫자 벡터가 아니라 다음과 같은 텍스트다.

```text
시스템 지시사항
+ 사용자의 원래 질문 문자열
+ 검색된 청크의 본문
+ 각 청크의 법령명·조문 번호·인용 ID
```

이 경계를 기억하면 임베딩 모델과 생성 모델의 역할이 섞이지 않는다.

### 저장과 재시도는 실패한 단계부터 수행한다

초기 설명에는 청킹 실패 시 파싱부터, 벡터 저장 실패 시 임베딩부터 다시 수행한다고 적었다. 앞 단계의 정상 산출물이 남아 있다면 불필요한 재계산이다.

```text
원본 저장 성공 → 파싱 실패       : 저장된 원본으로 파싱 재시도
파싱 성공 → 청킹 실패           : 파싱 결과로 청킹 재시도
청킹 성공 → 임베딩 실패         : 실패한 청크만 임베딩 재시도
임베딩 성공 → DB 저장 실패      : 계산된 벡터가 남아 있으면 저장만 재시도
```

이를 위해 각 산출물에는 원문 버전, 청킹 규칙 버전, 임베딩 모델과 버전, 처리 상태를 연결하는 것이 좋다. 같은 입력을 재시도해도 중복 행이 계속 생기지 않도록 안정적인 ID나 유일성 조건을 두는 것도 중요하다.

### 기술적 실패와 품질 실패를 분리한다

`문맥에 맞는 결과가 나오지 않음`은 여러 단계에 반복되어 있었지만 원인을 구분하기 어렵다.

- **기술적 실패:** 파서 예외, DB 쓰기 오류, 임베딩 API 실패, 벡터 차원 불일치처럼 요청이 정상 완료되지 않은 상태다.
- **검색 품질 실패:** 검색은 정상 실행됐지만 정답 근거 조문이 top-k에 들어오지 않은 상태다.
- **문맥 구성 실패:** 정답 청크를 검색했지만 잘리거나 중복되거나 인용 ID가 섞인 상태다.
- **생성 품질 실패:** 올바른 근거가 제공됐지만 모델이 근거 밖 내용을 말하거나 잘못 인용한 상태다.
- **표시 실패:** 답변과 인용 ID는 맞지만 화면의 조문명, 링크, 원문 연결이 잘못된 상태다.

검색과 답변을 별도로 평가해야 어느 경계를 고칠지 알 수 있다. 검색 실패인데 답변이 그럴듯한 경우도 성공으로 판정하면 안 된다.

### 메타데이터는 계층 표현보다 식별과 검증이 먼저다

장·절·조·항·호 계층을 남기는 것은 중요하다. 하지만 메타데이터의 직접적인 운영 목적은 검색된 텍스트가 어디에서 왔는지 안정적으로 식별하고 사용자에게 증명하는 것이다.

- **식별:** 같은 문구가 여러 법령에 있어도 법령명과 조문 번호로 구분한다.
- **인용:** 답변 아래에 `전기사업법 제7조 제1항`처럼 정확히 표시한다.
- **원문 확인:** 사용자가 링크를 열어 앞뒤 문맥과 예외를 직접 확인한다.
- **필터링:** 특정 법령, 시행 시점, 장·절로 검색 범위를 제한한다.
- **재현:** 어떤 버전의 어떤 조문이 당시 답변 근거였는지 추적한다.

배열의 순번이나 임시 `index 위치`는 데이터가 다시 적재되면 바뀔 수 있다. 문서 ID, 조문 ID, 법령명, 조·항·호, 원문 URL처럼 안정적인 식별 정보를 사용해야 한다.

## 법령 청킹 예시에 적용

학습자가 제공한 전기사업법 제7조 제1항은 두 문장을 함께 하나의 청크로 유지하는 것이 적절하다. `전기사업을 하려는 자`, `전기사업의 종류`, `산업통상자원부장관의 허가`처럼 구절로 나누면 주체·대상·의무가 분리되고, 변경허가에 관한 두 번째 문장도 첫 문장과의 관계를 잃는다.

```text
[전기사업법 > 제2장 > 제1절 > 제7조(사업의 허가) > 제1항]
전기사업을 하려는 자는 전기사업의 종류별로 산업통상자원부장관의 허가를 받아야 한다.
허가받은 사항 중 산업통상자원부령으로 정하는 중요 사항을 변경하려는 경우에도 또한 같다.
```

제5항처럼 여러 호가 긴 목록을 이루면 호별 청크를 고려할 수 있다. 이때 `재무능력 및 기술능력이 있을 것`만 저장하면 무엇의 기준인지 알 수 없으므로 상위 문장을 함께 붙인다.

```text
[전기사업법 제7조 제5항]
전기사업의 허가기준은 다음 각 호와 같다.

[제1호]
전기사업을 적정하게 수행하는 데 필요한 재무능력 및 기술능력이 있을 것.
```

첫 MVP의 규칙은 다음처럼 단순하게 시작한다.

1. 짧은 조문은 조 전체를 유지한다.
2. 여러 항이 있는 긴 조문은 항별로 나눈다.
3. 긴 호 목록은 상위 조건 문장을 붙여 호별로 나눈다.
4. 문장 내부의 구절이나 형태소 단위로는 자르지 않는다.
5. 실제 대표 질문에서 정답 조문이 top-k에 들어오는지 확인한 뒤 규칙을 바꾼다.

이 규칙은 현재의 설계 추천이며 아직 독립 실험으로 비교한 결과는 아니다.

## 코드와 AI 모델의 책임

현재 MVP에서 대부분의 파이프라인 제어는 코드와 DB가 담당한다. AI 모델을 사용하는 핵심 경계는 임베딩 생성과 답변 생성이다.

| 단계 | 기본 책임 | 기억할 이유 |
|---|---|---|
| 원문 파싱 | 코드 | 구조화된 HTML·XML·JSON의 조·항·호를 결정적으로 추출하고 같은 입력에 같은 결과를 낸다. |
| 청킹 | 코드 | 조·항·호 기반 규칙을 재현하고 테스트할 수 있어야 한다. |
| 문서 임베딩 | 임베딩 모델 | 청크 텍스트를 의미 비교가 가능한 벡터로 변환한다. |
| 벡터 저장 | 코드·DB | 청크 ID와 벡터, 모델 버전을 저장하고 인덱스를 관리한다. |
| 질문 임베딩 | 임베딩 모델 | 질문을 문서 청크와 같은 벡터 공간으로 변환한다. |
| 유사도 검색 | 코드·DB | 벡터 거리 또는 유사도를 계산해 top-k 순위를 반환한다. |
| 검색 문맥 구성 | 코드 | 검색 결과를 중복 제거하고 토큰 한도 안에서 인용 ID와 함께 조립한다. |
| 답변 생성 | 생성 모델 | 원래 질문과 검색된 텍스트 근거를 사용해 사용자용 답변을 만든다. |
| 근거 조문 표시 | 코드 | 모델이 참조한 인용 ID를 신뢰 가능한 DB 메타데이터와 링크로 변환한다. |

스캔 PDF처럼 구조가 없는 입력에는 OCR이나 문서 이해 모델이 보조적으로 필요할 수 있다. 그래도 조문 번호의 연속성, 누락, 중복은 코드로 검증해야 한다. 생성 모델에게 조문명이나 원문 URL을 자유롭게 만들게 하면 존재하지 않는 근거를 생성할 수 있으므로, 표시는 DB의 안정적인 메타데이터에서 가져온다.

## 기억해야 할 기준

1. **파싱은 구조 복원이고 청킹은 검색 경계 선택이다.** 두 단계 모두 텍스트를 나누는 것처럼 보여도 출력의 목적이 다르다.
2. **청크는 형태소가 아니라 독립적으로 읽을 수 있는 근거 단위다.** 작게 자르는 효율보다 의미와 인용 가능성을 먼저 본다.
3. **임베딩은 관련성 후보를 찾는다.** 벡터가 가깝다고 법적 정답이 보장되지는 않는다.
4. **질문 벡터는 검색용이고 원래 질문은 생성용이다.** 숫자 벡터를 답변 모델의 검색 문맥으로 넣지 않는다.
5. **검색 결과와 모델 답변은 별도의 산출물이다.** 검색 성공, 문맥 성공, 답변 충실도, 출처 정확성을 각각 판정한다.
6. **재시도는 마지막 정상 산출물 다음 단계부터 한다.** 모든 실패에서 전체 파이프라인을 다시 실행하지 않는다.
7. **출처는 모델이 만들어 내는 문자열이 아니다.** 안정적인 청크 ID와 DB 메타데이터를 코드가 연결한다.
8. **현재 청킹 규칙은 가설이다.** 대표 질문과 recall@k로 비교하기 전에는 좋은 규칙으로 확정하지 않는다.
