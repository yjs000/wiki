## 임베딩은 무엇을 숫자로 만드는가

임베딩(embedding)은 문장, 단어, 이미지 같은 대상을 여러 실수로 이루어진 벡터로 바꾼 결과다.

```text
"전기사업자는 허가를 받아야 한다"
    -> embedding model
    -> [0.013, -0.072, 0.118, ...]
```

텍스트 검색용 임베딩 모델은 관련 있는 텍스트가 벡터 공간에서도 가까워지도록 학습된다. 질문과 문서를 같은 모델로 임베딩한 뒤 벡터 관계를 비교하면, 표현이 완전히 같지 않아도 의미가 관련된 문서를 찾을 수 있다.

각 숫자 하나에 사람이 읽을 수 있는 이름이 붙는 것은 아니다. 첫 번째 좌표가 “허가”, 두 번째 좌표가 “사업”을 뜻하는 식으로 해석할 수 없는 경우가 대부분이다. 의미는 여러 좌표 사이의 관계에 분산되어 있다.

임베딩은 원문이나 정답도 아니다. 숫자, 부정어, 예외 조항처럼 중요한 차이를 놓칠 수 있다. 따라서 높은 임베딩 유사도는 두 문장이 법적으로 동일하거나 사실상 같은 결론이라는 증명이 아니다.

## 벡터와 차원은 무엇인가

벡터(vector)는 순서가 있는 숫자 목록이다. 기하학적으로는 원점에서 한 점까지 이어지는 화살표로 생각할 수 있다.

```text
[3, 4]                    2차원 벡터
[0.2, -0.7, 0.1]         3차원 벡터
[x1, x2, ..., x2048]     2048차원 벡터
```

차원(dimension)은 벡터에 들어 있는 숫자의 개수이자 표현 공간의 축 개수다. 2048차원 임베딩은 숫자가 2048개라는 뜻이다. 문장에 의미가 2048개 있거나 단어가 2048개라는 뜻은 아니다.

2차원에서는 가로축과 세로축을 그릴 수 있다. 2048차원은 사람이 그림으로 볼 수 없지만 길이, 각도, 거리 계산에는 같은 수학을 적용한다.

차원 수가 같다고 모든 벡터를 비교할 수 있는 것도 아니다. 서로 다른 임베딩 모델은 각자 다른 좌표계를 학습한다. 두 모델이 모두 512개 숫자를 반환해도 좌표의 의미가 맞지 않으므로 직접 비교해서는 안 된다. 질문에는 `query`, 검색 대상 문서에는 `passage`를 쓰도록 학습된 모델이라면 이 입력 구분도 일관되게 지켜야 한다.

## norm, 정규화, 재정규화

norm(노름)은 벡터의 크기를 재는 규칙이다. 임베딩에서 자주 쓰는 L2 norm은 원점에서 벡터 끝점까지의 직선거리다. 피타고라스 정리를 여러 차원으로 확장한 계산이다.

```text
||v||₂ = sqrt(v1² + v2² + ... + vn²)
```

`v=[3, 4]`라면 L2 norm은 `sqrt(3² + 4²) = 5`다. norm이 1인 벡터를 단위 벡터(unit vector)라고 한다. norm이 0인 영벡터는 방향이 없어서 코사인 유사도를 계산할 수 없다.

L2 정규화(normalization)는 모든 좌표를 벡터 자신의 L2 norm으로 나누는 연산이다.

```text
v = [3, 4]
||v||₂ = 5
normalize(v) = [3/5, 4/5] = [0.6, 0.8]
```

새 벡터의 길이는 `sqrt(0.6² + 0.8²) = 1`이다. 두 좌표 모두 같은 양수 `1/5`를 곱했으므로 `3:4`라는 비율과 화살표 방향은 그대로이고 길이만 바뀐다.

길이가 1보다 작을 때도 같다. norm이 `0.5`인 벡터를 `0.5`로 나누면 모든 좌표가 2배가 되어 길이가 1이 된다. 좌표를 각각 다르게 고치는 것이 아니라 전체를 같은 비율로 확대하는 것이다.

재정규화(renormalization)는 이미 정규화했던 벡터를 자르는 등 변형한 뒤 다시 정규화한다는 뜻이다. 별개의 수학 연산이 아니라 “변형 후 한 번 더 정규화”다.

## 내적과 코사인 유사도

내적(dot product)은 두 벡터의 같은 위치에 있는 숫자끼리 곱하고 모두 더한 값이다.

```text
a · b = a1*b1 + a2*b2 + ... + an*bn

[1, 2] · [3, 4] = 1*3 + 2*4 = 11
```

기하학적으로 내적에는 벡터의 길이와 두 화살표 사이 각도가 함께 들어 있다.

```text
a · b = ||a||₂ * ||b||₂ * cos(theta)
```

같은 방향의 성분은 양수로 더해지고 반대 방향의 성분은 음수로 상쇄된다. 하지만 방향이 같아도 벡터가 길수록 내적은 커진다. 방향만 비교하려면 길이의 영향을 제거해야 한다.

코사인(cosine)은 각도를 수치로 표현하는 함수다. 코사인 유사도는 내적을 두 벡터 길이의 곱으로 나눈다.

```text
cosine_similarity(a, b) = (a · b) / (||a||₂ * ||b||₂)
```

이렇게 하면 벡터 크기의 영향이 사라지고 방향 관계가 남는다.

- **1:** 같은 방향
- **0:** 직교, 즉 방향상 겹치는 성분이 없음
- **-1:** 정반대 방향

예를 들어 `a=[1, 0]`, `b=[1, 1]`이면 내적은 `1`, 두 norm은 `1`과 `sqrt(2)`이므로 코사인 유사도는 약 `0.707`이다.

두 벡터를 미리 L2 정규화했다면 두 norm이 모두 1이다. 이때는 분모가 1이므로 내적과 코사인 유사도가 같다.

```text
정규화된 a, b:  a · b = cosine_similarity(a, b)
```

코사인 유사도는 확률이 아니다. `0.9`는 90% 정답, 90% 동일, 90% 같은 법적 효력을 뜻하지 않는다. 실제 점수 분포는 모델과 데이터에 따라 다르다.

## 왜 임베딩 검색에 코사인을 쓰는가

임베딩 모델은 텍스트 관계가 벡터 공간의 상대적 위치와 방향에 나타나도록 학습된다. 코사인은 벡터의 절대 크기보다 방향을 비교하므로 의미적 관련성을 순위화하는 데 쓰기 좋다.

```text
질문 --같은 모델/query--> 질문 벡터
문서 --같은 모델/passage--> 문서 벡터들
질문 벡터와 각 문서 벡터의 cosine 계산
점수가 높은 문서부터 정렬
```

OpenAI는 자사 임베딩이 길이 1로 정규화되어 있고, 코사인 유사도를 권장한다고 설명한다. 단위 벡터끼리는 코사인 유사도를 내적으로 계산할 수 있어 계산도 단순해진다.

코사인이 모든 검색 문제에서 자동으로 최선이라는 뜻은 아니다. 모델 카드가 권장하는 유사도 함수와 입력 방식을 따라야 한다. 정확한 용어 일치가 중요한 검색에서는 키워드 검색을 함께 쓰고 실제 질의 평가셋으로 순위를 확인해야 한다.

## 2048차원을 512차원으로 자르는 경우

NVIDIA Nemotron 3 Embed 1B를 구체적인 사례로 보자. 이 모델의 native output은 2048차원이다. NVIDIA 공식 모델 카드는 첫 1024개 또는 첫 512개 좌표를 유지하는 축약을 지원하며, 자른 벡터를 similarity 계산 전에 L2 재정규화하라고 설명한다.

2048차원 벡터 `v`를 앞부분 `p` 512개와 뒷부분 `q` 1536개로 나누면 다음 관계가 성립한다.

```text
v = [p, q]
||v||₂² = ||p||₂² + ||q||₂²
```

전체 벡터가 단위 벡터여도 `q`에 0이 아닌 값이 있다면 `q`를 버린 뒤 `||p||₂`는 1보다 작다. 그래서 다음과 같이 재정규화한다.

```text
u = p / ||p||₂
```

여기에는 자주 섞이는 두 종류의 “방향”이 있다.

- **원래 2048차원 방향:** 유지되지 않는다. 뒤 1536개 좌표와 그 정보가 사라졌다. 2048차원 화살표와 512차원 화살표는 서로 다른 공간에 있다.
- **남은 512차원 안의 방향:** 재정규화해도 유지된다. `p`의 모든 좌표에 같은 양수 `1/||p||₂`를 곱하므로 좌표 사이 비율이 바뀌지 않는다.

재정규화는 삭제한 정보를 복원하지 않는다. 남아 있는 화살표의 길이만 다시 1로 맞춘다.

순수 코사인 공식은 계산 과정에서 norm으로 나누기 때문에, 같은 두 truncated vector의 코사인 값은 재정규화 전후에 수학적으로 같다. 그래도 재정규화가 필요한 이유는 모델 카드의 계약을 따르고, 저장 벡터의 norm을 1로 통일하며, 내적 기반 검색에서도 코사인과 같은 순위를 얻기 위해서다.

2048개 숫자를 512개로 줄이면 저장량과 단순 벡터 연산량은 약 4분의 1이 된다. 대신 삭제된 정보 때문에 검색 순위와 품질이 달라질 수 있다. 공식적으로 축약할 수 있다는 말은 2048차원과 품질이 완전히 같다는 뜻이 아니다.

### 왜 앞 512개인가

일반 임베딩 벡터에서 마지막 512개나 무작위 512개를 임의로 선택하면 안 된다. 어떤 좌표가 독립적으로 유용한지는 모델의 학습 방식과 출력 계약에 달려 있다. 이 사례에서 앞 512개를 쓰는 근거는 NVIDIA 모델 카드가 첫 1024개 또는 512개를 공식적으로 지원하기 때문이다.

Matryoshka Representation Learning은 짧은 앞부분도 독립적으로 유용하도록 nested representation을 학습하는 일반 방법을 보여 준다. 다만 NVIDIA 공개 모델 카드는 Nemotron 3 Embed 1B에 그 학습법을 사용했다고 명시하지 않는다. 이 논문은 prefix 축약이 가능한 모델의 일반 원리를 이해하기 위한 자료이고, 해당 NVIDIA 모델에 대한 직접 근거는 모델 카드다.

## 모델 카드와 API 옵션의 차이

모델 카드(model card)는 모델의 목적, 입력과 출력, 평가 조건, 적합한 용도와 한계를 설명하는 문서다. Model Cards 논문은 모델이 어떤 조건에서 평가됐고 어디에 쓰도록 설계됐는지 함께 보고하는 문서화 방식을 제안했다. 모델 카드는 API 자체가 아니라 모델 사용 설명서에 가깝다.

API는 배포된 모델을 네트워크로 호출하는 인터페이스다. 모델이 이론적으로 또는 공식적으로 지원하는 출력 방식과 현재 서버가 요청 옵션으로 구현한 기능은 서로 다를 수 있다.

```text
모델의 표현 능력과 출력 계약     -> 모델 카드
요청 필드, 허용 값과 오류 응답    -> API reference
실제 서비스에 필요한 변환        -> 클라이언트 adapter와 테스트
```

일부 모델 API는 `dimensions=512` 같은 요청을 받아 서버에서 짧은 벡터를 반환한다. OpenAI의 `text-embedding-3` 계열은 `dimensions` 매개변수로 기본 출력보다 짧은 임베딩을 요청할 수 있다.

NVIDIA Embedding NIM의 `dimensions` 지원 범위는 모델마다 다르다. 현재 API reference에서 `nvidia/nemotron-3-embed-1b`는 `dimensions`를 생략하거나 `2048`만 허용한다. `512`나 `1024`를 요청하면 HTTP 400 오류와 함께 허용값이 2048뿐이라고 응답한다. 하지만 모델 카드는 반환된 2048차원에서 첫 1024개 또는 512개를 유지할 수 있다고 설명한다.

따라서 이 조합에서는 다음 경계를 사용한다.

```text
NIM 요청: dimensions를 보내지 않음
NIM 응답: 2048차원
클라이언트: 응답 차원과 숫자 유효성 검증
클라이언트: 앞 512개 선택
클라이언트: L2 재정규화
최종 결과: 512차원 단위 벡터
```

어떤 서비스가 차원 옵션을 제공하고 다른 서비스가 제공하지 않는 이유는 모델 설계와 서빙 구현, 배포 버전의 지원 범위가 서로 다르기 때문이다. `dimensions`라는 필드가 있다는 사실만으로 임의의 숫자를 지원한다고 가정하면 안 된다. 모델 카드와 해당 API reference를 함께 확인해야 한다.

## 정리와 판단 기준

- **차원:** 벡터 좌표의 개수다. 의미의 개수가 아니다.
- **L2 norm:** 벡터 길이다. 각 좌표를 제곱해 더한 뒤 제곱근을 구한다.
- **정규화:** 모든 좌표를 같은 비율로 바꿔 norm을 1로 만든다.
- **내적:** 같은 위치의 값을 곱해 더한다. 방향과 길이의 영향을 모두 받는다.
- **코사인 유사도:** 내적을 두 norm으로 나눠 길이 영향을 제거하고 방향을 비교한다.
- **축약:** 정보를 버리는 차원 감소다. 모델이 공식 지원한 방법만 사용해야 한다.
- **재정규화:** 축약으로 길이가 달라진 벡터를 다시 단위 벡터로 만든다. 삭제된 정보는 복원하지 않는다.
- **모델 카드와 API:** 모델의 출력 계약과 서버가 구현한 요청 옵션은 구분해서 확인한다.

검색 시스템에서 임베딩을 사용할 때는 같은 모델과 같은 입력 규칙을 질문과 문서에 적용해야 한다. 차원, 유한값, 영벡터, norm을 검증하고, 코사인 점수를 확률로 해석하지 않아야 한다. 마지막으로 실제 도메인의 관련·비관련 문장으로 검색 순위를 평가해야 한다.
